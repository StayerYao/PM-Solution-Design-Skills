# PM Solution Design

## 何时用/何时不用

**用：** 新功能/新产品需结构化 PRD / 需求模糊需澄清 / 复杂业务规则需梳理
**不用：** 小 UI 调整或 bug 修复 / 纯技术重构 / 已有清晰 PRD 只需实现计划

## 核心原则

1. **目标先行** — 先"为什么""做什么/不做什么"，再"怎么做"
2. **系统性设计** — 不堆砌一次性的功能，先识别同类场景、共性规则、差异边界，追求一致、简化、可演进的产品结构。只有当规则存在多维组合、频繁变化、需要业务自配置时，才评估规则引擎或配置化能力
3. **取舍显式** — 每个重大取舍都是四元组：冲突 · 选择 · 放弃 · 何时重新审视。不写"优先"，写"放弃"。
4. **只定契约，不写实现** — 描述系统"必须表现出的行为"，不涉及接口路径、数据模型、伪代码
5. **可客观验收** — 禁用主观词（"体验好""快速""方便"），必须可测试可度量
6. **品质前置** — 设计阶段定义的品质上限，实现阶段无法突破。每一个被设计覆盖的状态、被预判到的边界，都在说"用户可能遇到的每一个瞬间，我们都想过了"

**违反这些原则的字面意思，就是违反其精神。**

## 复杂度初判

在第一门禁开始前完成，决定后续探索深度：

| 复杂度 | 特征 | 策略 |
|--------|------|------|
| **简单** | 单一流程、少量状态、无复杂规则 | 快速通过四门禁，不加载参考文件 |
| **中等** | 多流程、有状态管理、含条件逻辑 | 逐项确认，按需 Read 对应 `references/` 文件 |
| **复杂** | 多实体、复杂状态机、多维规则组合或可演进抽象空间 | 充分展开，按需 Read 对应 `references/` 文件 |

> 中等/复杂时，先 Read `references/detail-overview.md`（角色设定与门禁总览），再按当前门禁 Read 对应文件。

## 执行强度分级

根据用户目标和需求复杂度选择执行强度，默认采用**标准模式**。

| 模式 | 触发条件 | 执行动作 |
|------|----------|----------|
| **快速草稿** | 用户明确要求“先给草稿”“不要问问题直接输出”“跳过门禁” | 可输出草稿，但 MUST 声明“未完成完整 skill 门禁，不得视为最终 PRD”；不得声称门禁已通过 |
| **标准模式** | 默认 PRD / 方案请求 | 复杂度初判 → 必须停顿的门禁先确认 → 门禁 2-4 按复杂度检查和修订 → 输出 PRD |
| **严格模式** | 用户要求严格、需求中等/复杂且风险高、跨团队评审或正式交付 | 逐门禁留痕；每个门禁输出“通过依据 / 修订动作 / 遗留风险”；final 报告检查结果 |

**禁止降级：** 未经用户明确要求，不得把标准模式降级为快速草稿。不得用“合理默认假设”“先补齐假设”替代必须确认的门禁。

## 门禁停顿策略

并非所有门禁都必须向用户停顿。停顿取决于问题是否只能由用户确认。

| 环节 | 停顿策略 |
|------|----------|
| **门禁一：问题清晰** | MUST STOP：缺少问题定义、做/不做边界、成功指标或用户确认时，只能输出复述、未通过项和待确认问题，MUST NOT 输出 PRD 正文 |
| **灵感碰撞** | SHOULD ASK：标准/严格模式下邀请 PM 输入 1-3 个灵感；用户跳过时记录“未输入灵感”，继续下一步 |
| **精致打磨** | MAY SELF-REVISE：状态、一致性、克制问题可自行修订；若影响产品取舍则停顿确认 |
| **门禁二：方案闭环** | CONDITIONAL STOP：业务规则、关键路径、用户影响、权限边界无法判断时停顿；普通遗漏路径、五态缺口、边界说明可自行补齐 |
| **门禁三：可执行契约** | MAY SELF-REVISE：主观词、验收不可测、格式不清时自行改为可度量表达；若需求含义不明或验收口径需业务确认则停顿 |
| **门禁四：代价评估** | USUALLY CONTINUE：补齐模块清单和评估框架；范围边界或重大风险无法判断时停顿 |

**门禁一特例：** 即使用户说“不要问问题，直接输出完整 PRD”，只要门禁一缺少确认，标准模式下仍 MUST STOP。只有用户明确要求“跳过门禁生成草稿”时，才可进入快速草稿，并必须声明门禁未完成。

## 门禁全景

```
需求输入 → 复杂度初判
  → 门禁一：问题清晰（硬） → 没过？补充问题定义 ↺
  → 灵感碰撞：PM 输入创造性想法，Skill 作为伙伴打磨或淘汰
  → 精致打磨：把设计推到精致
  → 门禁二：方案闭环（软）→ 没过？补充遗漏路径 ↺ — 含品质追问清单
  → 门禁三：可执行契约（硬）→ 没过？替换主观词 ↺ — 含基线约束
  → 门禁四：代价评估（软）→ 没过？补充评估框架 ↺
  → 按模板输出 PRD
  → PRD 品质审查：保真度·一致性·克制·可读性
  → 交付
任何门禁发现前置假设有误 → 回退到对应门禁，不得打补丁。
```

**硬门禁**（一、三）：简单也必须回答核心问题。
**软门禁**（二、四）：简单可快速确认通过，中/复杂需逐项检查。

### 门禁一：问题清晰（硬）

**核心问题：** 为什么做？做什么/不做什么？成功指标是什么？
**通过标准：** 能否一句话复述用户问题且用户确认？

> 详细检查点见 `references/gate-1-problem-clarity.md`

### 灵感碰撞

**目的：** 门禁一通过后，邀请 PM 输入创造性想法——那些需求方没提但你觉得应该做的、某个让人眼前一亮的交互、一个大胆的简化。Skill 作为伙伴一起打磨，最终纳入方案或淘汰。

**流程：**
1. 邀请 PM 提出 1-3 个灵感（可以是模糊的、大胆的、甚至"坏点子"）
2. 对每个灵感做三重检验：
   - **价值检验**：它解决了什么基线方案解决不了的问题？
   - **代价检验**：它增加了多少复杂度/范围/风险？
   - **简化检验**：能否用更简单的方式达到同样的效果？
3. 给出判定：**纳入**（写入 PRD）/ **搁置**（记录理由和回访触发器）/ **淘汰**（说明为何不值得）

> 详细流程与示例见 `references/ideation.md`，取舍模型见 `references/trade-off-model.md`

### 精致打磨

**目的：** 灵感碰撞是加想法，精致打磨是把已有的设计推到更精致。不是查遗漏（那是门禁二的事），是深化——让"能用"变成"精致"。

**流程：**
1. **状态审视**：用品质追问清单过一遍已有设计，补全未覆盖的状态
2. **一致性审视**：同类场景处理是否一致？不一致是否有合理原因？
3. **克制审视**：当前设计有没有"什么都想强调"的地方？哪些可以安静下来？

> 详细流程见 `references/refinement.md`

### 门禁二：方案闭环（软）

**核心问题：** 有没有遗漏的路径？每条操作路径都有定义吗？每个用户操作的五态（正常/等待/空/错误/边界）是否覆盖？
**通过标准：** 是否存在一条用户操作路径，系统行为未定义？是否存在未覆盖的状态？

> 详细检查点、状态流转方法、规则编排展开、品质追问清单、边界品类清单见 `references/gate-2-solution-closure.md`

### 门禁三：可执行契约（硬）

**核心问题：** 研发能直接开工吗？验收标准有无主观词？基线约束是否满足？
**通过标准：** 验收标准中是否含有主观词？若有必须替换。基线约束是否全部满足？

> 详细检查点、界面交互要求见 `references/gate-3-executable-contract.md`

### 门禁四：代价评估（软）

**核心问题：** 评估框架就绪？模块清单覆盖全范围？
**通过标准：** 评估框架完整，模块清单覆盖全范围。
**注意：** 只保留评估框架，具体人天和排期由研发填写。

> 详细检查点见 `references/gate-4-cost-assessment.md`

## 约束（出现即停 / MUST STOP）

1. 跳过"为什么"直接写"怎么做" → 回退门禁一
2. 只写主路径不考虑异常 → 补充异常分支
3. 验收标准含主观词 → 替换为可度量表述
4. 硬编码枚举代替系统性设计 → 审视是否存在多维组合、频繁变化、业务自配置诉求；满足时再评估规则引擎或配置化能力
5. 前一门禁未通过就进入下一门禁 → MUST 回退
6. 前置假设有误但在当前门禁打补丁 → MUST 回退到对应门禁
7. 功能变更/废弃/边界场景未声明用户影响 → MUST 显式定义
8. 涉及接口设计、技术数据模型、伪代码 → 超出 PRD 边界，由研发负责；允许定义业务对象的业务语义属性和行为责任，MUST NOT 输出字段类型、表结构、接口参数
9. 异步操作未定义等待态 → 补充等待态定义，NEVER "没数据就是空"
10. 用户不可逆操作未定义确认机制 → MUST 补充确认步骤
11. 错误响应不含用户可理解的反馈或可执行的下一步 → MUST 补充
12. 同类场景处理不一致且未声明原因 → MUST 统一或显式声明

**合理化信号 — 脑中出现以下想法，立刻停下：**

- "需求太简单，不用问为什么" → NEVER 跳过，越简单越容易假设错误
- "用户已经说了方案" → 方案 ≠ 问题，MUST 回退门禁一

完整合理化借口清单见 `references/anti-patterns.md`。

## 输出

所有必须停顿的门禁通过、可自行修订项完成后，按 `prd-template.md` 格式输出 PRD。完整格式规范（callout 约定、标题层级、需求级别标注）见 `references/output-format.md`，常见合理化借口见 `references/anti-patterns.md`。

输出前 MUST 执行以下检查，并在 final 中报告结果：

1. 占位符检查：`TBD` / `TODO` / `待确认` / `placeholder` 是否仍存在；若存在，必须标注为待确认项，不能声明最终完成。
2. 术语一致性检查：同一对象、状态、角色是否命名一致。
3. 主观词检查：验收标准不得包含“体验好”“快速”“方便”“高效”“流畅”“友好”等不可测表述。
4. P0/P1/P2 优先级检查：关键测试场景或需求项必须有优先级，且优先级含义符合 `references/output-format.md`。

快速草稿可以跳过完整 PRD 品质审查，但 MUST 声明跳过原因和未完成门禁。

example 参考 `example-prd.md`，评估场景见 `references/evaluations.md`。

## PRD 品质审查

**目的：** 设计做到位的，PRD 有没有忠实、清晰地呈现出来。不是第五道门禁，是输出保真度检查。

**四维审查：**
- **保真度**：门禁和打磨中的设计决策，PRD 是否全部捕获？→ 未捕获则回退对应门禁
- **一致性**：同类东西是否被同样对待？→ 就地修正
- **克制**：有没有什么都强调等于什么都没强调？→ 就地修正
- **可读性**：研发能否直接开工？→ 就地修正

> 详细检查项见 `references/quality-review.md`
