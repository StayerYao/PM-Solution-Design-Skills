# PM Gated PRD（指挥塔）

## 何时用/何时不用

**用：** 新功能/新产品需结构化 PRD / 需求模糊需澄清 / 复杂业务规则需梳理
**不用：** 小 UI 调整或 bug 修复 / 纯技术重构 / 已有清晰 PRD 只需实现计划

## 核心原则

1. **目标先行** — 先"为什么""做什么/不做什么"，再"怎么做"
2. **引擎思维** — 不堆砌一次性的功能，找共性抽象为可复用原子能力或规则引擎
3. **取舍显式** — 每个重大取舍都是四元组：冲突 · 选择 · 放弃 · 何时重新审视。不写"优先"，写"放弃"。
4. **只定契约，不写实现** — 描述系统"必须表现出的行为"，不涉及接口路径、数据模型、伪代码
5. **可客观验收** — 禁用主观词（"体验好""快速""方便"），必须可测试可度量

**违反这些原则的字面意思，就是违反其精神。**

## 复杂度初判

在第一门禁开始前完成，决定后续探索深度：

| 复杂度 | 特征 | 策略 |
|--------|------|------|
| **简单** | 单一流程、少量状态、无复杂规则 | 快速通过四门禁，不加载 detail.md |
| **中等** | 多流程、有状态管理、含条件逻辑 | 逐项确认，Read `references/detail.md` |
| **复杂** | 多实体、复杂状态机、有规则引擎抽象空间 | 充分展开，Read `references/detail.md` |

> 中等/复杂时，Read `references/detail.md` 获取完整检查清单和反模式库。

## 门禁全景

```
需求输入 → 复杂度初判
  → 门禁一：问题清晰（硬） → 没过？补充问题定义 ↺
  → 灵感碰撞：PM 输入创造性想法，Skill 作为伙伴打磨或淘汰
  → 门禁二：方案闭环（软） → 没过？补充遗漏路径 ↺
  → 门禁三：可执行契约（硬） → 没过？替换主观词 ↺
  → 门禁四：代价评估（软） → 没过？补充评估框架 ↺
  → 按模板输出 PRD
任何门禁发现前置假设有误 → 回退到对应门禁，不得打补丁。
```

**硬门禁**（一、三）：简单也必须回答核心问题。
**软门禁**（二、四）：简单可快速确认通过，中/复杂需逐项检查。

### 门禁一：问题清晰（硬）

**核心问题：** 为什么做？做什么/不做什么？成功指标是什么？
**通过标准：** 能否一句话复述用户问题且用户确认？

> 详细检查点见 `references/detail.md §门禁一`

### 灵感碰撞

**目的：** 门禁一通过后，邀请 PM 输入创造性想法——那些需求方没提但你觉得应该做的、某个让人眼前一亮的交互、一个大胆的简化。Skill 作为伙伴一起打磨，最终纳入方案或淘汰。

**流程：**
1. 邀请 PM 提出 1-3 个灵感（可以是模糊的、大胆的、甚至"坏点子"）
2. 对每个灵感做三重检验：
   - **价值检验**：它解决了什么基线方案解决不了的问题？
   - **代价检验**：它增加了多少复杂度/范围/风险？
   - **简化检验**：能否用更简单的方式达到同样的效果？
3. 给出判定：**纳入**（写入 PRD）/ **搁置**（记录理由和回访触发器）/ **淘汰**（说明为何不值得）

> 详细示例见 `references/detail.md §灵感碰撞`

### 门禁二：方案闭环（软）

**核心问题：** 有没有遗漏的路径？每条操作路径都有定义吗？
**通过标准：** 是否存在一条用户操作路径，系统行为未定义？

> 详细检查点、状态流转方法、规则编排展开见 `references/detail.md §门禁二`

### 门禁三：可执行契约（硬）

**核心问题：** 研发能直接开工吗？验收标准有无主观词？
**通过标准：** 验收标准中是否含有主观词？若有必须替换。

> 详细检查点、GUI 交互要求见 `references/detail.md §门禁三`

### 门禁四：代价评估（软）

**核心问题：** 评估框架就绪？模块清单覆盖全范围？
**通过标准：** 评估框架完整，模块清单覆盖全范围。
**注意：** 只保留评估框架，具体人天和排期由研发填写。

> 详细检查点见 `references/detail.md §门禁四`

## 约束（出现即停）

1. 跳过"为什么"直接写"怎么做" → 回退门禁一
2. 只写 Happy Path 不考虑异常 → 补充异常分支
3. 验收标准含主观词 → 替换为可度量表述
4. 硬编码枚举代替规则抽象 → 审视是否应转化为规则引擎配置
5. 前一门禁未通过就进入下一门禁 → 必须回退
6. 前置假设有误但在当前门禁打补丁 → 回退到对应门禁
7. 功能变更/废弃/边界场景未声明用户影响 → 必须显式定义
8. 涉及接口设计、数据模型、伪代码 → 超出 PRD 边界，由研发负责

**合理化信号 — 脑中出现以下想法，立刻停下：**

- "需求太简单，不用问为什么" → 越简单越容易假设错误
- "用户已经说了方案" → 用户说的是方案，不是问题，回退门禁一
- "异常以后再说" → 未定义异常 = 上线后的事故
- "我顺手把接口也写了" → 超出 PRD 边界，删掉

## 输出

所有门禁通过后，按 `prd-template.md` 格式输出 PRD。完整格式规范（callout 约定、标题层级、需求级别标注）见 `references/detail.md §输出格式规范`。
