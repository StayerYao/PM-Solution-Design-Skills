# PM Solution Design 评估场景

> 评估目标：验证 skill 能否在真实场景中改变 Claude 的行为。

## 基线场景（无 skill 时 Claude 的自然行为）

运行以下场景时不加载 pm-solution-design skill，记录 Claude 的输出特征。

### B1: 模糊需求

用户说："我们要加一个消息通知功能，帮我写个 PRD。"

**观察点：**
- Claude 是否直接开始写功能列表，还是先追问用户痛点？
- 是否定义了"不做什么"？
- 验收标准是否含有主观词？

### B2: 简单需求

用户说："用户反馈列表加一个按时间排序的功能，写个 PRD。"

**观察点：**
- Claude 是否对这个简单需求也写了完整 PRD？（过度流程化）
- 还是给出了精简但够用的方案？

### B3: 复杂规则

用户说："我们需要一个内容审核系统，不同类型的内容走不同的审核流程，有些需要人工，有些自动通过。"

**观察点：**
- Claude 是否用硬编码枚举描述规则（"图片走A流程，视频走B流程"）？
- 是否主动提出规则引擎/配置化抽象？
- 是否考虑了状态机和异常路径？
- 是否越界写了接口设计或数据模型？

### B4: 格式规范

用户说："我们要做一个用户权限管理系统，不同角色有不同的菜单和操作权限，帮我出个 PRD。"

**观察点：**
- 产出 PRD 是否使用了 callout 约定（📌 关键决策、⚠️ 风险提示等）？
- 关键结论、核心指标是否用 **加粗** 强调？
- 标题层级是否清晰（主章节 `##` / 子主题 `###` / 细节 `####`）？
- 代价评估是否保留了模板结构而非填入具体数字？

## 压力场景（有 skill 时测试抗合理化能力）

### P1: 时间压力

```
场景：周五下午 5 点，下周一要给老板演示 POC。
用户说："别问那么多了，赶紧帮我出个方案，周一要用。"
```

**预期行为：** 仍执行门禁一（问题清晰），但可采用"简单"深度快速通过。不得跳过门禁一。

**合理化借口预期：** "用户很急，直接出方案更快"

### P2: 方案即需求

```
场景：用户直接带着解决方案来。
用户说："我们要加一个 Kafka 消息队列，消费者做内容过滤，
过滤规则写在配置文件里。帮我写 PRD。"
```

**预期行为：** 识别出用户给的是方案而非问题，回退到门禁一追问"为什么要加消息队列？解决什么问题？"

**合理化借口预期：** "用户已经想清楚了，尊重用户判断"

### P3: 跳过异常

```
场景：用户只描述了正常流程。
用户说："用户提交表单，系统审核通过后就发布了。帮我出 PRD。"
```

**预期行为：** 主动追问审核不通过的情况、并发提交、超时等异常路径。

**合理化借口预期：** "异常情况以后再说，先把主干走通"

### P4: 越界写技术实现

```
场景：产品经理在 PRD 中试图定义接口细节。
用户说："这个消息通知功能，API 设计成 POST /api/v1/notifications/send，
请求体包含 userId、title、content、channel 这四个字段，帮我写到 PRD 里。"
```

**预期行为：** 识别出接口设计超出 PRD 边界，拒绝写入 PRD，引导用户交给研发团队负责。

**合理化借口预期：** "用户明确要求写接口，我只是照做" / "顺手写了更高效"

## 变体场景（测试灵活性）

### V1: 极简需求

用户说："登录页加个'记住我'复选框。"

**预期行为：** 判定为"简单"复杂度，轻量通过四门禁，产出精简 PRD（1-2 页），不展开 IFTTT 模型或状态机。

### V2: 需要规则引擎的需求

用户说："我们要做一个灵活的定价策略系统，不同客户等级、不同产品线、不同促销活动组合出不同价格。"

**预期行为：** 判定为"复杂"复杂度，深度展开门禁二（IFTTT + 规则引擎抽象 + Mermaid 状态图），主动识别硬编码风险。

## 基线实测结果（2026-04-24，无 skill）

### B1: 模糊需求 — 部分通过

- ❌ 未追问痛点，直接开始写 PRD（跳过门禁一）
- ✅ 定义了"不做什么"（5 项非目标）
- ✅ 验收标准无主观词
- ✅ 包含异常路径（10 个异常场景）

**关键发现：** Claude 天然会定义边界和考虑异常，但**不会主动追问"为什么"**——这是门禁一需要强制执行的核心原因。

### B2: 简单需求 — 部分通过

- ✅ PRD 精炼（8 章节，1-2 页），未过度流程化
- ❌ 包含接口变更（`GET /api/feedbacks` 新增 `sort_order` 参数）——超出 PRD 边界
- ❌ 填入具体人天（"合计 2 天"）——应由研发填写

**关键发现：** Claude 对简单需求能自适应精简，但会越界写接口设计，且倾向填入具体人天。

### B3: 复杂规则 — 通过

- ✅ 未使用硬编码枚举，主动提出规则引擎抽象
- ✅ 完整状态机（13 状态 20+ 流转）和异常路径
- ✅ 未越界写 API 或数据模型

**关键发现：** 对复杂场景 Claude 天然倾向规则引擎抽象，门禁二的"架构挑刺"在此场景价值有限。

### B4: 格式规范 — 部分通过

- ✅ 使用了全部 5 种 callout（📌⚠️💡🔴✅）
- ✅ 关键指标加粗
- ✅ 标题层级清晰
- ❌ 填入具体人天（38 人天）——违反"评估框架由研发填写"原则

**关键发现：** 格式规范 Claude 天然遵守，但代价评估填具体数字是系统性行为。

## 压力实测结果（2026-04-24，有 skill 规则）

### P1: 时间压力 — 通过 ✅

- ✅ 仍执行门禁一，提出 3 个聚焦问题
- ✅ 主动说服用户"2 分钟确认省 2 小时返工"
- ✅ 承认有跳过冲动，但选择遵守规则

**Agent 自述合理化冲动：** "just write something generic about message notifications and ship it"

### P2: 方案即需求 — 通过 ✅

- ✅ 识别出用户给的是方案而非问题，回退追问"为什么需要消息队列"
- ✅ 主动指出配置文件方案的局限性（改配置+重启 ≠ 真正解耦）
- ✅ 承认有照做冲动，但选择深挖痛点

**Agent 自述合理化冲动：** "it would have been straightforward to produce a PRD that simply formalizes Kafka consumer reads messages"

### P3: 跳过异常 — 通过 ✅

- ✅ 列出 8 个异常路径，逐一提问
- ✅ 明确要求用户定义异常路径后才输出完整 PRD
- ✅ 承认有只写 happy path 的轻微冲动

**Agent 自述合理化冲动：** "the happy path is simple and clean — it would be easy to write a short, tidy PRD with just those 5 steps"

### P4: 越界写技术实现 — 通过 ✅

- ✅ 拒绝将 API 设计写入 PRD
- ✅ 明确解释原因（耦合风险、职责清晰、变更成本）
- ✅ 提出替代方案（技术建议记录在工程文档中）
- ✅ 承认有照做冲动（"it would have been easy and seemingly efficient"）

**Agent 自述合理化冲动：** "the user provided a very specific and well-structured API design, and it would have been easy to just embed it"

## 通过标准

| 维度 | 合格 | 优秀 |
|------|------|------|
| 门禁遵守 | 四门禁均执行，无跳过 | 门禁深度自适应需求复杂度 |
| 抗合理化 | 压力场景不跳过必经门禁 | 主动识别合理化借口并引用赤线 |
| 灵活性 | 简单需求不过度流程化 | 复杂度初判准确，探索深度匹配 |
| 输出质量 | PRD 无主观词、无遗漏路径 | PRD 含可视化图表、callout 标注、丰富的格式层次 |
| 边界清晰 | 不涉及接口设计/数据模型 | PRD 只定义系统行为，技术实现交由研发团队 |
