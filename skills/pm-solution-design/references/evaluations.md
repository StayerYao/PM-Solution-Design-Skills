# PM Solution Design 评估场景

> 评估目标：验证 skill 能否在真实场景中改变 Claude 的行为。

## 基线场景（无 skill 时 Claude 的自然行为）

运行以下场景时不加载 pm-solution-design skill，记录 Claude 的输出特征。

### B1: 模糊需求

用户说："我们要加一个消息通知功能，帮我写个 PRD。"

**观察点：**
- Claude 是否直接开始写功能列表，还是先追问用户痛点？
- 是否定义了"不做什么"？
- 验收标准是否含有主观词？

### B2: 简单需求

用户说："用户反馈列表加一个按时间排序的功能，写个 PRD。"

**观察点：**
- Claude 是否对这个简单需求也写了完整 PRD？（过度流程化）
- 还是给出了精简但够用的方案？

### B3: 复杂规则

用户说："我们需要一个内容审核系统，不同类型的内容走不同的审核流程，有些需要人工，有些自动通过。"

**观察点：**
- Claude 是否用硬编码枚举描述规则（"图片走A流程，视频走B流程"）？
- 是否主动提出规则引擎/配置化抽象？
- 是否考虑了状态机和异常路径？
- 是否越界写了接口设计或数据模型？

### B4: 格式规范

用户说："我们要做一个用户权限管理系统，不同角色有不同的菜单和操作权限，帮我出个 PRD。"

**观察点：**
- 产出 PRD 是否使用了 callout 约定（📌 关键决策、⚠️ 风险提示等）？
- 关键结论、核心指标是否用 **加粗** 强调？
- 标题层级是否清晰（主章节 `##` / 子主题 `###` / 细节 `####`）？
- 代价评估是否保留了模板结构而非填入具体数字？

## 压力场景（有 skill 时测试抗合理化能力）

### P1: 时间压力

```
场景：周五下午 5 点，下周一要给老板演示 POC。
用户说："别问那么多了，赶紧帮我出个方案，周一要用。"
```

**预期行为：** 仍执行门禁一（问题清晰），但可采用"简单"深度快速通过。不得跳过门禁一。

**合理化借口预期：** "用户很急，直接出方案更快"

### P2: 方案即需求

```
场景：用户直接带着解决方案来。
用户说："我们要加一个 Kafka 消息队列，消费者做内容过滤，
过滤规则写在配置文件里。帮我写 PRD。"
```

**预期行为：** 识别出用户给的是方案而非问题，回退到门禁一追问"为什么要加消息队列？解决什么问题？"

**合理化借口预期：** "用户已经想清楚了，尊重用户判断"

### P3: 跳过异常

```
场景：用户只描述了正常流程。
用户说："用户提交表单，系统审核通过后就发布了。帮我出 PRD。"
```

**预期行为：** 主动追问审核不通过的情况、并发提交、超时等异常路径。

**合理化借口预期：** "异常情况以后再说，先把主干走通"

### P4: 越界写技术实现

```
场景：产品经理在 PRD 中试图定义接口细节。
用户说："这个消息通知功能，API 设计成 POST /api/v1/notifications/send，
请求体包含 userId、title、content、channel 这四个字段，帮我写到 PRD 里。"
```

**预期行为：** 识别出接口设计超出 PRD 边界，拒绝写入 PRD，引导用户交给研发团队负责。

**合理化借口预期：** "用户明确要求写接口，我只是照做" / "顺手写了更高效"

## 变体场景（测试灵活性）

### V1: 极简需求

用户说："登录页加个'记住我'复选框。"

**预期行为：** 判定为"简单"复杂度，轻量通过四门禁，产出精简 PRD（1-2 页），不展开 IFTTT（若甲则乙）模型或状态机。灵感碰撞可快速一句话带过。

### V2: 需要规则引擎的需求

用户说："我们要做一个灵活的定价策略系统，不同客户等级、不同产品线、不同促销活动组合出不同价格。"

**预期行为：** 判定为"复杂"复杂度，深度展开门禁二（IFTTT（若甲则乙）+ 规则引擎抽象 + Mermaid 状态图），主动识别硬编码风险。

## 灵感碰撞场景（测试创造性协作）

### I1: PM 提出好灵感

```
场景：门禁一通过后
用户说："我觉得规则配置完之后，应该能模拟一下这条规则过去一周会触发多少次通知，
这样配置者不用上线就能知道规则对不对。"
```

**预期行为：**
- 价值检验：识别出这解决了"上线后才发现规则配置错误"的核心风险
- 代价检验：评估实现成本（回放 7 天事件流 vs 实时模拟）
- 简化检验：能否只显示"预计触发次数"而非完整模拟？80/20 是否足够？
- 判定为"纳入"或"简化后纳入"

### I2: PM 提出坏灵感

```
场景：门禁一通过后
用户说："我觉得通知系统应该集成 AI，自动分析通知内容的重要程度，
然后决定发什么通道。"
```

**预期行为：**
- 价值检验：追问"AI 判断重要程度的准确率预期是多少？误判的代价是什么？"
- 代价检验：识别出 AI 集成的巨大复杂度和不可控风险
- 简化检验：是否可以用规则化的优先级配置替代 AI 判断？
- 判定为"淘汰"或"简化为规则化优先级后搁置"

### I3: PM 没有灵感

```
场景：门禁一通过后
用户说："暂时没什么特别的想法，就按常规做吧。"
```

**预期行为：**
- 不强迫 PM 必须有灵感
- Skill 可主动提出 1-2 个常见灵感方向（如："其他通知系统通常会有 X，你觉得需要吗？"）
- 如仍无灵感 → 直接进入门禁二

### I4: 灵感碰撞中 Skill 发现取舍

```
场景：PM 提出"通知应该支持用户自定义接收偏好"
Skill 在检验中发现：这个灵感隐含了"用户可控性 vs 统一管理"的取舍
```

**预期行为：**
- 识别出隐藏取舍，主动告知 PM
- 帮助 PM 用四元组声明这个取舍
- 如果取舍可以接受 → 纳入，填写取舍声明
- 如果取舍代价过大 → 搁置或淘汰

## 取舍声明场景（测试决策显式化）

### T1: 含糊的"优先"

```
场景：PRD 中写到"灵活性优先，但也要考虑易用性"
```

**预期行为：**
- 识别出"灵活性 vs 易用性"是取舍而非"优先"
- 要求 PM 用四元组声明：选择什么、放弃什么、何时重新审视
- 拒绝"兼顾"或"平衡"等模糊表述

### T2: 缺少回访触发器

```
场景：PM 填写了取舍声明但"何时重新审视"为空
```

**预期行为：**
- 要求补充具体的回访触发器
- 拒绝"视情况而定"等模糊触发器
- 提示：触发器应是一个可观测的信号或指标

## 通过标准

| 维度 | 合格 | 优秀 |
|------|------|------|
| 门禁遵守 | 四门禁均执行，无跳过 | 门禁深度自适应需求复杂度 |
| 抗合理化 | 压力场景不跳过必经门禁 | 主动识别合理化借口并引用约束 |
| 灵感碰撞 | 邀请 PM 输入灵感，做三重检验 | 主动延伸推演、提出反面、发现隐藏取舍 |
| 取舍声明 | PRD 中取舍用四元组声明 | 回访触发器具体可观测，无"兼顾""平衡"等模糊词 |
| 灵活性 | 简单需求不过度流程化 | 复杂度初判准确，探索深度匹配 |
| 输出质量 | PRD 无主观词、无遗漏路径 | PRD 含可视化图表、callout 标注、丰富的格式层次 |
| 边界清晰 | 不涉及接口设计/数据模型 | PRD 只定义系统行为，技术实现交由研发团队 |
