# 常见合理化借口

| 借口 | 现实 |
|------|------|
| "需求太简单，不用深究问题" | 简单需求最易因假设错误返工 |
| "用户已经说了要什么" | 用户说的通常是方案，不是问题 |
| "异常情况以后再说" | 未定义异常 = 上线后的生产事故 |
| "硬编码先上线，以后再抽象" | 以后永远不会来 |
| "门禁太重了，先跳过" | 跳过省的时间会在实现阶段加倍还回 |
| "这个主观词大家都懂" | "大家都懂" = 每个人理解不同 |
| "功能变了用户自己会适应" | 未声明的行为变更 = 生产事故，严禁静默处理 |
| "接口设计我先顺手写了" | 超出 PRD 边界，应由研发团队负责 |
| "加载中不用特别处理" | 没有等待态 = 用户以为卡死了 |
| "这种错误一般不会发生" | 未定义的异常 = 上线后的事故 |
| "这个跟那个差不多，处理方式显而易见" | "显而易见" = 两个人理解不同 |
| "先这样，体验细节后面再打磨" | 设计阶段的品质上限锁死后无法突破 |
