# Output Scope

本文件定义 PRD 内容范围交互。用户可以决定输出包含或不包含哪些模块，但不能删除有效 PRD 的底线。

## 询问方式

在 G1 之后、G2 之前询问：

```text
这次输出更偏向哪种用途？
1. 正式研发交付：完整 PRD。
2. 方案评审：方案 Brief。
3. 创意探索：轻量方案。
4. 快速草稿：记录假设和待确认项。

是否需要特别包含或排除这些模块：状态流转、用户故事、界面交互、异常流程与响应、场景泛化、业务规则设计、权限边界、埋点与观测、代价评估？
```

## 不可省略底线

任何输出至少保留：

- 问题与目标。
- 做 / 不做边界。
- 成功指标或待确认成功指标。
- 核心流程。
- 验收标准或验收口径待确认项。
- 未确认项。

## 默认模块

完整 PRD 默认包含：问题与目标、边界、成功指标、取舍、场景泛化、业务对象、状态流转、业务规则设计、用户故事、界面交互、异常流程与响应、验收标准、关键测试场景、观测指标、风险和代价评估框架。

方案 Brief 默认包含：问题与目标、边界、核心流程、关键状态摘要、关键业务规则摘要、取舍、风险、待确认项。

快速草稿默认包含：已知信息、合理假设、待确认项、核心流程草案、不能视为最终 PRD 的声明。

## 裁剪规则

- 用户可以不展开完整状态图，但不能不定义关键状态。
- 用户可以不写详细界面交互，但涉及界面操作时必须保留正常流程、异常流程和用户可见响应。
- 用户可以不写完整用户故事，但正式研发交付必须保留主路径和关键异常分支摘要。
- 用户可以不写完整代价评估，但正式评审必须保留模块影响和风险框架。
- 用户要求删除异常流程时，改为精简异常摘要，并说明风险。
