# Quality Review

品质审查发生在输出组装后，用于确认文档忠实呈现门禁结论。它不是新门禁，而是输出保真度检查。

## 按复杂度执行

简单需求：检查保真度和可读性。确认 G1 问题、边界、成功指标和最小验收标准没有丢失。

中等需求：在简单检查基础上，增加一致性和克制审查。确认核心状态、异常响应、用户故事、验收标准和关键测试场景一致。

复杂需求：执行完整四维审查。确认 G1-G4 的关键结论全部进入输出，五态覆盖、边界品类、业务规则设计、执行基线、风险和未确认项没有遗漏。

## 四维审查

保真度：门禁和实践环节形成的关键决策是否进入输出。问题定义丢失时回退 G1；遗漏行为定义时回退 G2；遗漏执行契约时回退 G3。

一致性：同类角色、状态、操作、错误响应和术语是否一致。不一致时就地修订；如果不一致来自业务规则冲突，回退 G2。

克制：输出是否围绕目标和边界展开，是否把可选增强写成必须项。过度展开时收敛为摘要；如果收敛影响做 / 不做边界，回退 G1。

可读性：研发、测试、设计和业务评审是否能按文档继续推进。信息层级不清时重组输出；验收不可执行时回退 G3。

## 回退规则

| 问题 | 处理 |
|---|---|
| 问题、目标、边界、成功指标缺失 | 回退 G1 |
| 核心流程、状态、五态覆盖、边界品类、业务规则设计缺失 | 回退 G2 |
| 用户故事、界面交互、执行基线、验收标准、测试场景缺失 | 回退 G3 |
| 输出模式、风险、未确认项、模块影响范围缺失 | 回退 G4 |
| 格式、标题、术语轻微不一致 | 就地修订 |

## 必检项

- 无未标注的 `TBD`、`TODO`、`待确认`、`placeholder`。
- 同一对象、状态、角色、指标命名一致。
- 验收标准无“体验好”“快速”“方便”“高效”“流畅”“友好”等主观词。
- 关键测试场景有 P0/P1/P2 优先级。
- 未出现 API、数据模型、伪代码或具体人天。
