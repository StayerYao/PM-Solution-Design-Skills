# PRD Contract

完整 PRD 是研发、测试、设计和业务评审可共同使用的产品行为契约。它不写技术实现。

## 产品判断层

必须包含：背景与问题、目标用户、目标与成功指标、做 / 不做边界、关键取舍、场景泛化结论。

场景泛化结论应说明本需求是定制功能、通用能力，还是可演进规则空间。

## 方案闭环层

必须包含：核心业务对象、核心流程、状态流转、五态覆盖、边界品类、业务规则设计、异常流程与响应、权限和边界。

状态流转必须说明关键状态的进入条件、退出条件和失败路径。五态覆盖必须检查正常态、等待态、空态、错误态、边界态。边界品类必须按复杂度检查显示边界、数据边界、状态边界、权限边界、兼容边界。复杂需求使用 Mermaid 状态图。

业务规则设计必须说明触发、条件、动作。存在多维组合、频繁变化或业务自配置诉求时，评估配置化或规则抽象。

## 执行契约层

必须包含：用户故事、界面交互、执行基线、验收标准、关键测试场景、必要的埋点与观测。

用户故事覆盖主路径和关键异常分支。界面交互覆盖正常流程、异常流程和用户可见响应。异步操作必须定义等待态，不可逆操作必须定义确认机制，错误响应必须包含用户可执行下一步。验收标准必须客观可测。

## 评审治理层

必须包含：门禁状态、模块影响范围、风险与未确认项、代价评估框架、品质审查结果。

代价评估只保留模块和风险框架，不填写具体人天或排期。

## 禁止内容

不得包含 API 路径、请求体、响应体、字段类型、表结构、主键、外键、索引、伪代码、具体人天。
