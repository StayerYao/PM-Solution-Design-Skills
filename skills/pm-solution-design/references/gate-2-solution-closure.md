# 门禁二：方案闭环（软门禁）

**目标**：系统行为无遗漏路径——每条路都有定义的终点。

**检查点：**
- [ ] 核心业务对象已识别，且只定义业务语义属性和行为责任，未写成字段类型、表结构或接口参数
- [ ] 关键业务规则已定义
- [ ] 非功能约束底线已明确
- [ ] 全状态覆盖已确认（品质追问清单）
- [ ] 边界品类已逐项检查

**按深度展开：**

| 条目 | 简单 | 中等 | 复杂 |
|------|------|------|------|
| 状态流转 | 口头描述关键状态 | 文字列出状态流转 | Mermaid 状态图 + 退出机制 |
| 规则编排 | 直接描述规则 | IFTTT（若甲则乙）拆解 | IFTTT（若甲则乙）+ 系统性抽象评估 |
| 非功能约束 | 一句话底线 | 按维度列出 | 分维度 + 量化阈值 |

**系统性挑刺**：发现硬编码枚举（"参数为 A 则执行 B"的穷举）时，先判断是否存在多维组合、频繁变化、业务自配置诉求；满足时再主动反问是否应抽象为规则引擎或配置化能力。

## 业务对象边界

产品侧需要定义对象的业务语义，不定义技术数据模型。

| 允许 | 禁止 |
|------|------|
| 对象在业务中的含义 | 数据库表名 |
| 用户/业务必须理解的属性 | 字段类型、主键、外键、索引 |
| 对象参与的规则、状态、流程 | 接口字段、请求体、响应体 |

示例：允许写"通知包含接收人、触达渠道、内容主题、发送状态等业务属性"；禁止写 `userId`、`channel: enum`、`created_at`、`POST /api/xxx`、`notifications` 表。

## 品质追问清单

对每个用户可触发的操作，逐项确认五态覆盖：

| 操作 | 正常态 | 等待态 | 空态 | 错误区 | 边界区 |
|------|--------|--------|------|--------|--------|
| [操作1] | [预期响应] | [异步时用户看到什么] | [没数据时看到什么] | [失败时看到什么+下一步] | [极端输入怎么处理] |

- **等待态**：异步操作加载中用户看到什么？不允许"没数据就是空"
- **空态**：列表为空、搜索无结果、无权限时用户看到什么？
- **错误区**：失败时用户看到什么？能否理解？有无下一步行动？
- **边界区**：极端输入/极端数据量，系统怎么处理？

按复杂度分级：
- 简单：口头确认核心操作的正常态和错误区
- 中等：对核心操作逐项填写五态
- 复杂：对所有用户操作逐项填写五态，附 Mermaid 状态图

## 边界品类清单

按品类逐一过，而非开放式"想想还有没有"：

| 品类 | 检查项 |
|------|--------|
| 显示边界 | 长文本截断、极端数值、极宽/极窄屏幕 |
| 数据边界 | 空数据、超大数据量、并发写入、重复提交 |
| 状态边界 | 网络中断、操作超时、会话过期 |
| 权限边界 | 无权限访问、角色切换、数据越权 |
| 兼容边界 | 多语言、暗色模式、辅助模式（如有） |

按复杂度分级：
- 简单：只看数据边界和状态边界
- 中等：看数据边界 + 状态边界 + 显示边界
- 复杂：全部品类
