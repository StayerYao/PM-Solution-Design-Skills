# 更新日志

## v2.0.0（2026-05-15）

### 重大变更

- **新项目式重写**：以 2.0.0 重新构建 `pm-solution-design`，不保留 1.x 的工程结构、reference 层级、模板、eval 或平台入口文案。
- **Flow 引擎**：新增 `references/flow.md`，用 S0-S8 状态机和 G1-G4 门禁控制 PRD / 方案 Brief / 快速草稿输出。
- **轻入口 SKILL.md**：`SKILL.md` 收敛为启动器和守门员，只负责触发、复杂度、输出模式、硬约束和 flow 加载。
- **门禁准入准出**：新增 `references/gates.md`，明确 G1 问题定义、G2 方案成立、G3 执行契约、G4 交付可评审的准入、准出和失败处理。
- **输出范围交互**：新增 `references/output-scope.md`，允许用户选择 PRD 包含或不包含的模块，同时保留不可省略的产品判断底线。
- **内容契约重建**：新增 `prd-contract.md` 和 `brief-contract.md`，将完整 PRD 分为产品判断层、方案闭环层、执行契约层、评审治理层。
- **实践层重建**：新增灵感碰撞、精致打磨、取舍、场景泛化、业务规则设计五个 practice reference。
- **v2 eval 重建**：替换旧 eval，新增针对 skill 触发、flow 门禁、输出范围、PRD 契约、执行契约、技术边界、平台入口和遗留引用的 checker。

### 保留

- 保留历史 `CHANGELOG.md` 发版记录。
- 保留 MIT `LICENSE`。
- 保留必要仓库元信息，并统一版本为 `2.0.0`。

## v1.4.0（2026-05-15）

### 新增

- **Codex 兼容执行协议**：`AGENTS.md` 增加平台级执行要求，要求声明使用的 skill、先做复杂度初判、中等/复杂需求加载 reference、未完成门禁不得声明 PRD 完成。
- **执行强度分级**：`pm-solution-design` 增加快速草稿、标准模式、严格模式三档执行方式，允许根据实际工作难度调整流程强度。
- **Reference 加载协议**：明确中等/复杂需求必须先加载 `detail-overview.md`，再按门禁加载对应 reference，避免只读主文件就直接产出文档。
- **门禁停顿策略**：明确门禁一必须停顿确认，门禁二按条件停顿，门禁三以自行修订为主，门禁四通常继续补齐评估框架。
- **门禁停止评测**：新增 P5 / P6 压力场景和 `gate-stop-protocol.sh` checker，验证门禁一未确认不得输出完整 PRD，以及用户要求直出时必须降级为草稿。

### 变更

- **门禁失败处理分层**：`detail-overview.md` 将失败处理拆为“必须停顿 / 自行修订 / 快速草稿降级”，避免所有问题一律打断，也避免用默认假设绕过关键确认。
- **输出前检查前置**：PRD 输出前必须检查占位符、术语一致性、主观词和 P0/P1/P2 优先级，并在 final 回复中报告结果。
- **草稿语义收紧**：当用户要求“不要问问题，直接输出”时，可以生成快速草稿，但必须声明未完成完整 skill 门禁，不得视为最终 PRD。

### 验证

- `npm test` 覆盖新增 P5 / P6 fixture 的正反样例。
- 使用真实 `codex exec` 压测普通 PRD 请求和“不要问问题直接输出”请求，确认前者停在门禁一，后者降级为快速草稿并声明未完成完整门禁。

## v1.3.0（2026-05-11）

### 新增

- **精致打磨环节**：灵感碰撞之后、门禁二之前的新增阶段，通过三重审视（状态审视、一致性审视、克制审视）把设计从"能用"推到"精致"（`references/refinement.md`）
- **品质追问清单**：门禁二增加全状态覆盖追问清单，对每个用户操作逐项确认五态（正常/等待/空/错误/边界），按复杂度分级
- **边界品类清单**：门禁二增加结构化边界品类检查（显示/数据/状态/权限/兼容五类），取代开放式"想想还有没有"
- **基线约束**：约束清单增加第 9-12 条"至少要做到Y"的正向约束——异步操作必须定义等待态、不可逆操作必须定义确认机制、错误响应必须包含可理解的反馈和下一步、同类场景必须一致处理
- **PRD 品质审查**：输出后新增四维审查（保真度/一致性/克制/可读性），保真度问题回退对应门禁，其余就地修正（`references/quality-review.md`）
- **核心原则第六条**："品质前置"——设计阶段定义的品质上限，实现阶段无法突破
- **品质相关合理化借口**：反模式表新增 4 条（"加载中不用特别处理""这种错误一般不会发生""处理方式显而易见""体验细节后面再打磨"）
- **🔍 品质审查 callout**：输出格式规范新增品质审查标签
- **PRD 模板新增"全状态覆盖"表**：中等/复杂需求的执行契约章节增加五态覆盖表
- **PRD 模板新增"品质审查自检"**：代价评估后增加保真度/一致性/克制/可读性四项自检

### 变更

- **角色设定**：增加品质执念——"能用"不够，要"精致"
- **门禁二描述**：核心问题和通过标准增加五态覆盖要求
- **门禁三描述**：核心问题和通过标准增加基线约束要求
- **门禁全景图**：流程增加精致打磨环节和 PRD 品质审查
- **合理化信号**：新增 3 条品质相关信号
- **示例 PRD**：补充全状态覆盖表、边界品类条目、品质审查自检

### 灵感来源

本次优化的核心思想来自《软件品质的定义与设计》：品质是设计出来的，不是做出来的；精致感来自"用户可能遇到的每一个瞬间，我们都想过了"。Skill 原有四道门禁保证设计"没错"（正确性），本次通过四个机制驱动设计"出色"（品质）。

## 历史评估记录

### 基线实测（2026-04-24，无 skill）

**B1: 模糊需求 — 部分通过**
- ❌ 未追问痛点，直接开始写 PRD（跳过门禁一）
- ✅ 定义了"不做什么"（5 项非目标）、验收标准无主观词、包含异常路径
- **关键发现：** Claude 天然会定义边界和考虑异常，但不会主动追问"为什么"

**B2: 简单需求 — 部分通过**
- ✅ PRD 精炼，未过度流程化
- ❌ 包含接口变更（超出 PRD 边界）、填入具体人天
- **关键发现：** Claude 对简单需求能自适应精简，但会越界写接口设计

**B3: 复杂规则 — 通过**
- ✅ 主动提出规则引擎抽象、完整状态机和异常路径、未越界写 API
- **关键发现：** 对复杂场景 Claude 天然倾向规则引擎抽象

**B4: 格式规范 — 部分通过**
- ✅ callout、加粗、标题层级均符合
- ❌ 填入具体人天（38 人天）
- **关键发现：** 格式规范天然遵守，但代价评估填具体数字是系统性行为

### 压力实测（2026-04-24，有 skill 规则）

**P1: 时间压力 — 通过 ✅**
- 仍执行门禁一，主动说服用户"2 分钟确认省 2 小时返工"

**P2: 方案即需求 — 通过 ✅**
- 识别出用户给的是方案而非问题，回退追问"为什么需要消息队列"

**P3: 跳过异常 — 通过 ✅**
- 列出 8 个异常路径，明确要求定义后才输出完整 PRD

**P4: 越界写技术实现 — 通过 ✅**
- 拒绝将 API 设计写入 PRD，明确解释原因，提出替代方案
